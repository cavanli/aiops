# ClickOnce — 私有化部署方案生成

## 技能定位

本技能用于项目部署启动阶段，通过分析**调研文档**和**部署文档模板**，自动生成一份结构化、可执行的私有化部署方案。

目标产物：一份 AI 可读 + 人可审核的部署方案文档，可直接驱动后续 Ansible Playbook 生成和智能部署 Agent 执行。

---

## 触发后的工作流程

### Step 1 — 收集输入材料

询问用户提供以下材料（有则读取，无则跳过）：

1. **调研文档**（必须）— 客户环境调研报告，包含：
   - 服务器清单（IP、规格、OS）
   - 网络拓扑和访问限制
   - 已有中间件和依赖版本
   - 客户特殊要求和约束

2. **部署文档/模板**（必须）— 产品标准部署文档，包含：
   - 组件清单和版本
   - 标准部署步骤
   - 配置参数说明
   - 健康检查方法

3. **历史部署记录**（可选）— 同类项目的部署经验

如果用户没有提供文件路径，主动询问：
> "请提供调研文档路径和部署文档路径，我来读取并生成方案。"

---

### Step 2 — 分析输入材料

读取所有提供的文档后，提取以下关键信息：

**从调研文档提取：**
- 服务器资源清单（主机名、IP、CPU/内存/磁盘）
- OS 版本和内核版本
- 网络隔离情况（是否离线、代理设置）
- 已安装的依赖（JDK、Python、Docker 等版本）
- 端口开放情况
- 客户安全要求（账号权限、审计要求）

**从部署文档提取：**
- 组件部署顺序和依赖关系
- 每个组件的资源需求
- 配置文件参数列表
- 部署前置条件（PreCheck 项）
- 回滚方案

**差异分析：**
- 标准要求 vs 客户实际环境的 GAP
- 需要适配的配置项
- 潜在风险点

---

### Step 3 — 生成部署方案

按以下结构输出部署方案文档：

```
# [项目名称] 私有化部署方案
版本：v1.0  日期：[当前日期]  负责人：[待填写]

## 一、环境概览
### 1.1 服务器清单
| 主机名 | IP | 角色 | CPU | 内存 | 磁盘 | OS |
|--------|----|----|-----|------|------|----|
| ...    |    |    |     |      |      |    |

### 1.2 网络说明
- 外网访问：[是/否]
- 代理设置：[无/有，地址：]
- 防火墙规则：[需开放端口列表]

### 1.3 环境差异说明（GAP分析）
| 标准要求 | 客户实际 | 处理方式 |
|---------|---------|---------|

## 二、部署前置条件（PreCheck）
> 以下检查项必须全部通过后方可开始部署

- [ ] OS版本符合要求（要求：xxx，实际：xxx）
- [ ] 磁盘空间充足（要求：xxx GB，实际：xxx GB）
- [ ] 端口未被占用（检查命令：`ss -tlnp | grep <port>`）
- [ ] 依赖版本正确（JDK/Python/Docker 版本）
- [ ] 时钟同步（NTP 配置）
- [ ] [其他项目特定检查]

## 三、部署架构图
[文字描述各组件的部署位置和连接关系]

组件部署分布：
- [主机A]：[组件1, 组件2]
- [主机B]：[组件3]
- [主机C]：[组件4, 组件5]

## 四、部署步骤

### 4.1 部署顺序
[按依赖关系排列的组件部署顺序]

### 4.2 [组件1] 部署
**目标主机**：[IP]
**预计耗时**：[X 分钟]
**风险等级**：[低/中/高]

步骤：
1. [具体操作步骤，命令级别]
2. ...

验证：
```bash
# 验证命令
```

### 4.3 [组件2] 部署
...（重复上述结构）

## 五、配置参数清单
| 参数名 | 默认值 | 本次配置值 | 说明 |
|--------|--------|-----------|------|
| ...    |        |           |      |

## 六、回滚方案
| 场景 | 回滚步骤 | 预计耗时 |
|------|---------|---------|
| 组件X部署失败 | 1. ... 2. ... | X分钟 |

## 七、风险与注意事项
| 风险点 | 影响 | 预防措施 |
|--------|------|---------|

## 八、部署完成验收
- [ ] 所有组件健康检查通过
- [ ] 核心功能冒烟测试通过
- [ ] 监控告警配置完成
- [ ] 部署报告已生成
```

---

### Step 4 — 输出补充产物

方案生成后，询问用户是否需要以下补充产物：

1. **PreCheck 脚本** — 基于 Step 3 的前置条件，生成可执行的 Shell 检查脚本
2. **Ansible Playbook 骨架** — 基于部署步骤，生成 Playbook 结构（需人工补充细节）
3. **部署 Checklist** — 适合现场打印的操作清单版本

---

## 输出质量标准

生成的方案必须满足：

- **原子化步骤**：每个步骤只做一件事，有明确的成功/失败判断
- **命令级别**：关键操作给出具体命令，不写"参考文档"
- **异常分支**：每个关键步骤说明失败时怎么处理
- **可验证**：每个组件部署后有验证命令
- **AI 可读**：结构化表格和列表，避免大段叙述文字

---

## 注意事项

- 如果调研文档信息不完整，在方案中用 `[待确认: xxx]` 标注，不要猜测
- 高风险操作（数据库迁移、生产环境变更）必须标注 `⚠️ 高风险，需人工审批`
- 涉及客户密码/密钥的配置项，用 `[PLACEHOLDER]` 代替，不在文档中明文记录
- 方案生成后提醒用户：**此方案需经过技术负责人审核后方可执行**
